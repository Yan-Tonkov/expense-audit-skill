Ответь на вопрос пользователя по таблицам менеджеров, используя индекс `sheets-index.md`. Перед поиском сам проверяй, не устарел ли индекс, и при необходимости пересобирай его — не полагайся на то, что кто-то заранее запустил `/index-sheets`.

## 1. Вопрос пользователя

Аргумент: `$ARGUMENTS`

Если вопрос не передан — спроси пользователя, что он хочет узнать по таблицам.

## 2. Проверь свежесть индекса

У таблиц нет вебхуков или другого способа узнать об изменениях заранее, поэтому свежесть индекса проверяется по времени его сборки (TTL):

```bash
node scripts/index-status.js --file sheets-index.md
```

Первая строка вывода — одна из:
- `MISSING` — файла `sheets-index.md` ещё нет.
- `STALE ...` — индекс старше порога (по умолчанию 7 дней) или дату сборки не удалось прочитать.
- `FRESH ...` — индекс в пределах порога, можно использовать как есть.

Если `MISSING` или `STALE` — перестрой индекс тем же способом, что делает скилл `/index-sheets`:

```bash
node scripts/sheets-build-index.js
```

Сообщи пользователю коротко, что индекс был устаревшим/отсутствовал и был пересобран (это займёт ~20-30 секунд), и продолжай уже со свежим `sheets-index.md`. Если `FRESH` — просто иди дальше.

## 3. Прочитай индекс

Прочитай файл `sheets-index.md` в корне репозитория. Это список всех 10 таблиц: заголовок (обычно имя менеджера), URL (уже с нужным `gid`) и список колонок каждой таблицы. У каждого менеджера — свой формат: разные названия колонок, разный состав данных (сделки, лиды, звонки, активные клиенты).

## 4. Выбери 1–3 релевантные таблицы по колонкам

В индексе нет содержательных описаний (только колонки и заголовок — по природе табличных данных им неоткуда взяться), поэтому выбирай по:
- **Заголовку таблицы** — часто это имя менеджера и тип данных (например `ksenia-calls` — звонки Ксении, `oleg-clients-2025` — клиенты Олега, `katya-mql` — маркетинговые лиды Кати, `petrov-clients` — клиенты Петрова).
- **Составу колонок** — есть ли `MRR`/`Сумма`/`Тариф` (сделки/выручка), `Статус`/`Этап`/`Стадия` (воронка), `Телефон`/`Что обсуждали` (звонки), `Риски` (проблемные сделки) и т.д.
- Если вопрос называет конкретного менеджера по имени — сопоставь имя с заголовком таблицы (заголовки образованы от имён: `artem-` → Артём, `lena-` → Лена и т.д.).
- Если вопрос про конкретную компанию/клиента, а не менеджера — компания может встречаться в нескольких таблицах сразу (менеджеры иногда ведут одного и того же клиента параллельно или передают друг другу) — в этом случае стоит открыть больше кандидатов (до 3), а не одну.

Не открывай больше 3 таблиц на первой итерации.

## 5. Открой только выбранные таблицы

```bash
node scripts/sheets-fetch.js <номер1> <номер2> ...
```

Аргумент — номер таблицы (1–10, как в заголовках `sheets-index.md`), либо spreadsheetId, либо полный URL. Можно передать несколько за один вызов. Скрипт выводит для каждой таблицы её URL и полное содержимое как Markdown-таблицу с дополнительной колонкой **«Строка»** — это реальный номер строки в исходном листе Google Sheets.

Если открытые таблицы не содержат ответа — вернись к индексу (шаг 4) и выбери другого кандидата. Не более 2–3 таких итераций — если ответа всё ещё нет, честно скажи, что не нашёл, чем перебирать все 10 таблиц подряд.

## 6. Сформулируй ответ

- Отвечай по существу, на языке вопроса.
- Приводи конкретные факты из найденных строк (компания, сумма, статус, дата и т.д.), а не общие фразы.
- **Источник каждого факта — ссылка на таблицу и номер строки**, в формате `{URL из шага 5}&range=A{N}:Z{N}`, где `{N}` — значение из колонки «Строка» найденной записи. Например: `https://docs.google.com/spreadsheets/d/.../edit?gid=118287130&range=A5:Z5`.
- Если один и тот же клиент/компания встречается в нескольких таблицах с разными (возможно противоречащими) данными — это нормально для этого набора данных (разные менеджеры ведут своё видение одной сделки). Не пытайся выбрать "правильную" версию сама — покажи обе, процитируй обе строки и укажи, что данные разошлись между менеджерами.
- Если ответа в таблицах нет — прямо скажи об этом, не придумывай факты.
